01
Functional Dependency
Definition
A functional dependency X → Y means: for any two tuples t1, t2 in relation R, if t1[X] = t2[X] then t1[Y] = t2[Y].
"Same value in X always implies same value in Y"
Determinant
X in X → Y
The left-hand side (X) that determines
Dependent
Y in X → Y
The right-hand side (Y) that is determined
Types of Functional Dependency
TypeConditionExample
Trivial FD Y ⊆ X (Y is a subset of X) RollNo,Name → Name
Non-Trivial FD X ∩ Y = Φ (nothing in common) RollNo → Name
Partial FD Proper subset of CK → NPA Ssn → Ename in (Ssn,Pnum) CK
Full FD Removing any LHS attr breaks dependency {Ssn,Pnum} → Hours
Transitive FD NPA → NPA (chain through non-prime attrs) B → C where A→B (A is CK)
⚡
Exam Trick: To check if X → Y holds — look for two rows with the same X value. If they have different Y values, the FD does NOT hold.
02
Armstrong's Axioms (IR1–IR6)

Six inference rules to derive new FDs from existing ones. IR1–IR3 are the primary (Armstrong's) axioms; IR4–IR6 are derived.

IR1 — Reflexive Rule
If Y ⊆ X, then X → Y
A,B → A (always true, trivial)
IR2 — Transitivity
X→Y, Y→Z ⟹ X→Z
Name→Marks, Marks→Dept ⟹ Name→Dept
IR3 — Augmentation
X→Y ⟹ XZ → YZ
RollNo→Name ⟹ RollNo,Marks→Name,Marks
IR4 — Decomposition
X→YZ ⟹ X→Y and X→Z
A→BC ⟹ A→B and A→C
IR5 — Union / Additive
X→Y, X→Z ⟹ X→YZ
A→B, A→C ⟹ A→BC
IR6 — Pseudo-Transitivity
X→Y, WY→Z ⟹ WX→Z
A→B, CB→D ⟹ CA→D
✓
Remember: Decomposition applies to RHS only. You cannot decompose LHS. AB→C cannot become A→C or B→C.
03
Closures & Equivalence
Closure of an Attribute Set X (written X⁺)
The set of all attributes that can be functionally determined by X using the given FD set F.
X⁺ = all attributes Y such that X → Y can be derived from F
Algorithm: Computing X⁺
1
Start: result = {X} (X itself, by Reflexivity)
2
For each FD A → B in F: if A ⊆ result, add B to result
3
Repeat Step 2 until no new attributes are added
4
result = X⁺
Example — R(A,B,C,D,E), F = {A→B, B→C, C→D, D→E}
A⁺ = {A} →add B (A→B)→ {A,B} →add C (B→C)→ {A,B,C} →add D (C→D)→ {A,B,C,D} →add E (D→E)→ {A,B,C,D,E} CD⁺ = {C,D} →add E (D→E)→ {C,D,E} ← C,D,E only!
Equivalence of FD Sets (E ≡ F)

Two FD sets E and F are equivalent if E covers F AND F covers E.

E covers F
For every FD X→Y in F, compute X⁺ with respect to E. If X⁺ contains Y, then E covers that FD.
Example — E = {A→B, AB→C, D→AC, D→E}, F = {A→BC, D→AE}
Check E covers F:
A→BC: compute A⁺ wrt E = {A,B,C} ✓ (contains B and C) D→AE: compute D⁺ wrt E = {D,A,C,E,B} ✓ (contains A and E) Check F covers E:
A→B: A⁺ wrt F = {A,B,C} ✓ AB→C: AB⁺ wrt F = {A,B,C} ✓ D→AC: D⁺ wrt F = {D,A,E,B,C} ✓ D→E: D⁺ wrt F = {D,A,E,B,C} ✓ ∴ E ≡ F (equivalent)
04
Minimal Cover (Canonical Cover)
Definition
A minimal cover F' of a set F has no redundant attributes on either side and no redundant FDs. It is equivalent to F.
3-Step Algorithm
1
Decompose RHS — Every FD must have a single attribute on RHS.
A→BC becomes A→B and A→C
2
Remove extraneous (redundant) LHS attributes — For each FD AB→C, check if A→C or B→C already holds. If yes, remove the redundant attribute. Also remove any trivial FDs.
3
Remove redundant FDs — Using transitivity, if an FD can be derived from others, remove it.
If B→D and D→A exist, then B→A is redundant.
Example 1 — E = {B→A, A→D, AB→D}
Step 1: RHS already single. No change. Step 2: AB→D — check if A→D exists. YES! So B is extraneous. Replace AB→D with B→D E' = {B→A, A→D, B→D} Step 3: B→A and A→D ⟹ B→D (by transitivity). So B→D is redundant. Remove B→D. Minimal Cover = {B→A, A→D}
Example 2 — F = {A→B, C→B, D→ABC, AC→D}
Step 1: D→ABC becomes D→A, D→B, D→C F = {A→B, C→B, D→A, D→B, D→C, AC→D} Step 2: AC→D — check A→D? No. Check C→D? No. Keep AC→D. Step 3: D→A and A→B ⟹ D→B is redundant. Remove D→B. Minimal Cover = {A→B, C→B, D→A, D→C, AC→D}
⚠
Common Mistake: Do Step 1 FIRST before checking LHS redundancy. Doing it out of order gives wrong results.
05
Keys & Attributes
Key TypeDefinitionExample
Super Key Set of attributes uniquely identifying every tuple. Redundant attrs allowed. {BookId}, {BookId,Name}, {BookId,Name,Author}
Candidate Key (CK) Minimal super key — no proper subset is a super key. If {A} is SK and A alone determines all, it's a CK
Primary Key One chosen candidate key designated for the relation. Arbitrarily picked from CKs
Foreign Key Attribute(s) referencing the primary key of another relation. Dept.Dno references Department.Dno
Prime Attribute Member of some candidate key. If CKs = {AD, BD}, then A,B,D are prime
Non-Prime Attribute (NPA) Not a member of any candidate key. All remaining attributes
Finding Candidate Keys — Algorithm
1
Find (ABCD…)⁺ — it equals all attributes (confirms it's a superkey)
2
Remove attributes whose RHS appears in some FD. Keep reducing the SK.
3
Check: is the reduced SK still a super key? (compute its closure)
4
Check subsets — if no proper subset is a SK, it's a Candidate Key
5
Check if any prime attribute appears in RHS of FDs → if yes, there may be more CKs. Replace prime attrs and repeat.
Example — R(A,B,C,D), F = {A→B, B→C, C→A}
Step 1: (ABCD)⁺ = {A,B,C,D} ✓ SK = {ABCD} Step 2: A→B so remove B; A,B→C so remove C; SK = {AD} Step 3: AD⁺ = {A,D,B,C} ✓ still SK Step 4: A⁺={A,B,C} not SK; D⁺={D} not SK → {AD} is CK Step 5: A is in RHS of C→A → try {CD}: C⁺={C,A,B} not SK; D⁺={D} not SK → {CD} is CK C is in RHS of B→C → try {BD}: B⁺={B,C,A} not SK; D⁺={D} not SK → {BD} is CK Candidate Keys = {AD, CD, BD} Prime Attributes = {A, B, C, D} (all!)

1NF — First Normal Form
Rule
Every attribute must contain only atomic (indivisible) values. No multi-valued or composite attributes.

How to fix violations:

  • Method 1: Move the multi-valued attribute + primary key into a separate table
  • Method 2: Expand the key — create one row per value
  • Method 3: Create separate columns for each possible value (Dlocation1, Dlocation2…)
2NF — Second Normal Form
Two Conditions
1. Must be in 1NF
2. No Partial Dependency: no proper subset of any Candidate Key → Non-Prime Attribute

Violation pattern: Proper subset of CK → NPA

Check FDs for 2NF Violation
FDType2NF Status
CK → NPAFull CK determines NPAOK
Subset of CK → NPAPartial DependencyVIOLATES
NPA → NPANPA determines NPAOK for 2NF
Decomposition Rule
For each violating FD A→B where A is a subset of CK: create a new table R2(A,B) and remove B from the original table R1.
3NF — Third Normal Form
Definition 1 (Intuitive)
1. Must be in 2NF
2. No Transitive Dependency: NPA → NPA is not allowed
Definition 2 (Formal — use this for all checks)
For every non-trivial FD X→Y in R, either:
• X is a Super Key, OR
• Y is a Prime Attribute

BCNF — Boyce-Codd Normal Form (Stricter than 3NF)
Definition
For every non-trivial FD X→Y in R, X must be a Super Key.
There is NO exception for prime attributes (unlike 3NF)
⚡
Key Difference: Every relation in BCNF is in 3NF, but NOT every 3NF relation is in BCNF.
3NF allows NPA → PA but BCNF does NOT.

Lossless Join & Dependency Preserving Decomposition
Lossless Join (LJ)
When decomposed tables R1 and R2 are re-joined, the result is exactly the original R — no spurious (extra fake) tuples appear.
Dependency Preserving (DP)
All original FDs can be checked in individual decomposed tables without needing a join. Every FD in F is covered by some Ri.
Test for Lossless Join (2 tables)
Sufficient Condition
Decomposition of R into R1 and R2 is lossless if:
• (R1 ∩ R2) → R1, OR
• (R1 ∩ R2) → R2
i.e., the common attributes form a super key for at least one of R1 or R2
Test for Dependency Preservation
1
For each FD X→Y in F, check if X and Y both appear in the same Ri
2
If every FD is "covered" by some Ri, decomposition is dependency preserving
3
If some FD spans multiple tables (can't be checked without join), DP is violated
⚡
BCNF decomposition always achieves Lossless Join but may lose Dependency Preservation. 3NF decomposition can achieve both. This is why 3NF is sometimes preferred over BCNF.
08
Database Anomalies
Insertion Anomaly
Cannot insert data without having other (unrelated) data present first. E.g., can't add a new dept without adding an employee.
Update Anomaly
Changing one fact requires updating multiple rows. Missing one causes inconsistency. E.g., dept building stored repeatedly.
Deletion Anomaly
Deleting a row causes unintended loss of other information. E.g., deleting last employee in a dept erases dept info.

Transaction Management
Transaction
A logical unit of work — a sequence of database operations (reads/writes) that must be executed as a single unit. Either ALL operations succeed (committed) or NONE do (aborted/rolled back).
ACID Properties
PropertyMeaningEnsures
Atomicity All-or-nothing execution No partial updates. If T fails midway, all changes are undone (rollback).
Consistency Database remains in a valid state Transaction moves DB from one consistent state to another. Integrity constraints maintained.
Isolation Concurrent transactions don't interfere Partial results of one T are invisible to others. Achieved via concurrency control (locks).
Durability Committed changes persist Once committed, even a system crash can't undo the changes (write-ahead logging).
Transaction States
Active
→ Transaction is executing
Partially Committed
→ Last statement executed, not yet written to disk
Committed
→ Changes written to disk permanently ✓
Failed
→ Error occurred
Aborted
→ Rolled back, DB restored to state before T started
Schedules
Schedule TypeDefinitionSafe?
Serial Schedule Transactions execute one after another (no interleaving) Always correct
Serializable Schedule Equivalent result to some serial schedule, but may interleave Correct
Conflict Serializable Can be transformed to a serial schedule by swapping non-conflicting ops Correct
Non-Serializable Cannot be equivalent to any serial schedule May cause errors
Conflicting Operations
Two operations conflict if: (1) They belong to different transactions, (2) They access the same data item, and (3) At least one is a Write.
Read-Read: No conflict. Read-Write or Write-Write: Conflict.
Concurrency Problems
ProblemDescriptionExample
Dirty Read T2 reads a value written by T1 before T1 commits. If T1 aborts, T2 has wrong data. T1 writes X=50, T2 reads X=50, T1 aborts → T2 used ghost value
Lost Update Two transactions read and update the same data. One update is overwritten by the other. T1 reads X=100, T2 reads X=100, T1 writes X=150, T2 writes X=120 → T1's update lost
Unrepeatable Read T1 reads same data twice and gets different values because T2 updated it in between. T1 reads X=100, T2 updates X=200 and commits, T1 reads X=200 → inconsistent
Phantom Read T1 re-executes a query and gets more/fewer rows because T2 inserted/deleted rows. T1 counts rows = 5, T2 inserts a row, T1 counts again = 6 → phantom row

Locking (Concurrency Control)
Lock
A mechanism to control concurrent access to data items. A transaction must acquire a lock on a data item before accessing it.
Types of Locks
Shared Lock (S-Lock / Read Lock)
Acquired for reading. Multiple transactions can hold S-lock on the same item simultaneously.
lock-S(X) — allows read(X)
Exclusive Lock (X-Lock / Write Lock)
Acquired for writing. Only ONE transaction can hold X-lock. No other lock (S or X) can coexist.
lock-X(X) — allows read(X) and write(X)
Lock Compatibility Matrix
T2 requests S-Lock T2 requests X-Lock
T1 holds S-Lock ✓ Compatible ✗ Wait
T1 holds X-Lock ✗ Wait ✗ Wait
Two-Phase Locking (2PL)
Protocol
Every transaction must follow two phases:
Phase 1 — Growing Phase: Acquire locks only; no releasing allowed.
Phase 2 — Shrinking Phase: Release locks only; no new locks allowed.
✓
2PL guarantees conflict serializability. The point where a transaction acquires its last lock is called the Lock Point. The serial order of transactions = order of their lock points.
Strict 2PL
All exclusive (write) locks are held until the transaction commits or aborts. Prevents dirty reads. Avoids cascading rollbacks.
Rigorous 2PL
All locks (both S and X) held until commit/abort. Strictest form. Transactions appear to execute serially.
Deadlock
Definition
A situation where two or more transactions are waiting for each other to release locks — creating a cycle of waiting with no transaction able to proceed.
Prevention
Wait-Die scheme: Older T waits, younger T aborts.
Wound-Wait scheme: Older T aborts younger T, younger T waits.
Detection
Build a Wait-For Graph. If the graph has a cycle → deadlock exists. Choose a victim transaction to abort.
Classic Deadlock Example
T1: lock-X(A) → waiting for lock-X(B) T2: lock-X(B) → waiting for lock-X(A) ↑ Neither can proceed — DEADLOCK!
Lock Granularity
GranularityUnit LockedConcurrencyOverhead
FineTuple / RecordHighHigh

CoarseTable / DatabaseLowLow

Intention Locks (IS, IX, SIX) — used in multi-level locking to indicate that a finer-grained lock will be acquired inside. Prevents conflicts without checking all lower levels.

NoSQL Databases & Key-Value Stores ⭐ IMPORTANT
NoSQL ("Not Only SQL")
A class of database systems that provide a mechanism for storage and retrieval of data that is modeled differently from relational tables. Designed for large-scale, distributed systems with high availability and horizontal scalability.
CAP Theorem (Brewer's Theorem)

A distributed system can only guarantee two of these three properties at any time:

C — Consistency
Every read receives the most recent write or an error. All nodes see the same data at the same time.
A — Availability
Every request receives a (non-error) response, even without the latest data. System is always operational.
P — Partition Tolerance
System continues operating even if network splits occur between nodes (messages dropped/delayed).
⚡
In real distributed systems, P (Partition Tolerance) must always be chosen because network failures always happen. So the real choice is between CP (consistent but may be unavailable) vs AP (available but may be inconsistent).
NoSQL Database Types
🗝
Key-Value Store
Simplest NoSQL model. Data stored as {key: value} pairs. Extremely fast lookups. Examples: Redis, DynamoDB, Riak
📋
Document Store
Data stored as documents (JSON/BSON). Flexible schema. Examples: MongoDB, CouchDB, Firestore
📊
Column-Family
Data stored in column families, optimized for large-scale analytics. Examples: Cassandra, HBase, BigTable
🕸
Graph Database
Data stored as nodes and edges for relationship-heavy data. Examples: Neo4j, Amazon Neptune, ArangoDB
⭐ KEY-VALUE STORE — Deep Dive
Definition
A Key-Value Store is the simplest form of NoSQL database. It stores data as a collection of key-value pairs, similar to a dictionary or hash map. The key is a unique identifier; the value can be any data (string, JSON, binary, lists, sets, etc.).
Core Operations
OperationSyntaxDescription
GET GET(key) Retrieve value by key. O(1) average time
PUT / SET PUT(key, value) Insert or update a key-value pair
DELETE DELETE(key) Remove a key-value pair
EXISTS EXISTS(key) Check if a key exists. Returns boolean
Internal Structure — Hash Table Based
Key → Hash Function → Bucket/Slot → Value "user1" → h("user1") → Slot 42 → {name:"Alice", age:25} "user2" → h("user2") → Slot 17 → {name:"Bob", age:30} "sess1" → h("sess1") → Slot 99 → "token_abc123"

The hash function maps each key to a specific memory location, enabling O(1) average case lookups regardless of dataset size.

Data Partitioning — Consistent Hashing
Consistent Hashing
In distributed KV stores, data is distributed across multiple nodes using a hash ring. Each key is hashed and assigned to the nearest node on the ring in clockwise direction. When a node is added/removed, only K/N keys need to be moved (K = total keys, N = nodes) — minimizes data migration.
Ring Example
Ring: 0 ────── Node A (25) ──── Node B (60) ──── Node C (90) ──── 100 Key hash = 40 → goes to Node B (nearest clockwise) Key hash = 70 → goes to Node C Key hash = 95 → wraps around → goes to Node A
Replication in Key-Value Stores

To ensure availability and fault tolerance, data is replicated across multiple nodes.

TermMeaning
Replication Factor (N)Number of copies of each data item stored
Write Quorum (W)Minimum nodes that must confirm a write succeeds
Read Quorum (R)Minimum nodes to read from for a consistent result
Quorum ConditionR + W > N ensures strong consistency
Quorum Example with N=3
N=3, W=2, R=2 → R+W=4 > 3 ✓ → Strong Consistency N=3, W=1, R=1 → R+W=2 < 3 → Eventual Consistency (faster, less safe) N=3, W=3, R=1 → R+W=4 > 3 ✓ → Very strong but slow writes
Versioning & Conflict Resolution
Vector Clocks
Each node maintains a logical clock. Used to detect and order concurrent updates. Helps resolve version conflicts without requiring a centralized timestamp.
Eventual Consistency
If no new updates are made, eventually all replicas will converge to the same value. Used by AP systems (availability + partition tolerance).
Redis — Most Important KV Store

Redis (Remote Dictionary Server) is an in-memory key-value store supporting multiple value types.

Redis Persistence Options
RDB (Redis Database): Periodic snapshots to disk (fast restart, possible data loss).
AOF (Append-Only File): Logs every write command (slower but durable).
Hybrid: Both RDB + AOF for best balance.
Advantages & Disadvantages
Advantages
  • Extremely fast — O(1) lookups via hash
  • Simple data model — easy to use
  • Highly scalable — horizontal scaling
  • Schema-less — flexible value types
  • High availability via replication
  • In-memory = sub-millisecond latency
Disadvantages
  • No complex queries (no SQL JOIN, GROUP BY)
  • No relationships between keys
  • Not suitable for complex data models
  • Limited transaction support (no full ACID)
  • Secondary index support is limited
  • Memory cost for large datasets
Use Cases for Key-Value Stores
Session Management
Store user session data (auth tokens) with TTL (time-to-live). Fast lookups per request.
Caching
Cache DB query results, HTML fragments, API responses. Reduces DB load dramatically.
Real-time Leaderboards
Redis Sorted Sets for instant rank lookups in games, apps.
Shopping Cart
Cart data per user as a hash. Very fast add/remove items.
Pub/Sub Messaging
Redis supports publish/subscribe pattern for real-time messaging.
Rate Limiting
Count API requests per user using atomic increment with TTL.

BASE Properties (NoSQL's answer to ACID)
PropertyMeaning
Basically Available System guarantees availability — reads and writes always work, though possibly stale
